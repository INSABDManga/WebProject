Rapport de conception

Réalisation du projet

Le club BD/Manga est une association de l’INSA Lyon. Le club est propriétaire d’une bibliothèque de plus de 5200 ouvrages, en lecture libre pour tous. Pour le projet de Web dynamique nous avons décidé de mettre en place un site web afin de faire connaître cette association.

Les objectifs que nous avions défini étaient les suivants :

  • Un site durable (site stable, bien documenté et facile d’entretien)
  • Vitrine (site joli et ergonomique avec présentation, actu, info pratiques…)
  • Outil : moteur de recherche des ouvrages disponibles au club
  • Flux RSS pour suivre les actualités du club
  • Responsive design
  • Outil : WebApp (mettre notre outil de gestion de bibliothèque en ligne)
  • Interactions entre utilisateurs : système de notation/recommandation des ouvrages

Nous avons choisi d’aborder le projet dans une démarche agile. C’est-à-dire que les objectifs cités précédemment sont classés par ordre de priorité. Nous les avons suivi dans l’ordre et un à un. De plus, dans cette méthode la vision du client est très importante. Nous nous sommes donc référés à Léo, qui est le président du club BD/Manga, pour avoir son avis sur ce que l’on faisait. Nous avons travaillé avec Git pour ce projet. Lorsque l’objectif en cours était terminé et accepté par Léo, nous l’assemblions avec le projet global. Le fait de ne pas travailler directement sur le projet global nous permettait de garder une sauvegarde fonctionnelle au cas où un problème se produirait lors de la réalisation des nouvelles parties. La méthode agile nous permet aussi de définir des objectifs mais de ne pas forcément tous les réaliser. En effet, nous nous sommes plutôt attardé sur la qualité au lieu de la quantité. Nous pouvons donc ainsi livrer un site web fonctionnel avec les objectifs les plus importants réalisés : la durabilité du site, la vitrine et l’outil de moteur de recherche des ouvrages.

Les autres objectifs pourront donc être réalisés plus tard si nous voulons améliorer le site web. En effet, comme nous avons créé un site durable, les membres du club pourront reprendre notre code et ainsi l’améliorer selon leurs envies.

Organisation du groupe

Pour réaliser le projet nous avons tout d’abord suivi des tutoriels afin de se former aux différents outils que nous allions utiliser. Ensuite, nous nous sommes répartis les différentes parties du site : Back-end/Front-end. A partir de là nous nous sommes spécialisés dans notre domaine tout en restant connectés aux autres parties du projet pour rester cohérent dans ce que nous faisions.

Lors du développement des différentes parties nous avons fait des tests pour être sûr que tout fonctionnait bien. Par exemple, pour le formulaire de contact nous avons envoyé plusieurs mails, essayé toutes les fonctionnalités (bon captcha, mauvais captcha, etc…) afin de voir si le mail ne s’envoyait bien que lorsque tout était correctement renseigné. Ensuite, nous avons fait de même pour les commentaires et avons envoyé une vingtaine de commentaires, plus ou moins long pour différents articles. Nous avons vérifié l’affichage des commentaires et le dynamisme du formulaire. Un autre exemple de test, pour la pagination dynamique sur les pages “Actus” et “Le festival” nous avons créé plus d’une vingtaine d’articles pour voir si le changement des pages se faisait correctement.

Fonctionnement de Django

Django utilise une variante du modèle MVC : le MVT (Modèle, View, Template).

Tout d’abord, comme pour n’importe quel site web, le serveur écoute et accepte les requêtes de connexion. Ensuite, il envoie la requête au contrôleur. Celui-ci fait notamment le mapping url et aiguille vers une vue. Les vues sont le centre de la fabrication des pages web en utilisant les modèles et les templates. Les modèles sont un système haut niveau de gestion de données. Les templates permettent la mise en forme de la page. Ils sont écrits en HTML, CSS et JS. Voici donc un exemple de fonctionnement d’une vue :

  • prise en paramètre des infos de l’url
  • recherche ou modification de la base de données via les modèles
  • mise à disposition de variables auprès du template
  • renvoie du template encapsulé dans une requête HTML

Notons qu’une vue peut interagir avec d’autres éléments, tels que les formulaires ou des méthodes contenues dans des packages extérieurs.

 

II/Fonctionnement général

1/Vitrine

a/Modèles

Dans cette partie, le schéma représente la base de donnée tandis que la liste représente les modèles. Pour développer sur ce site web, seulement la liste de modèles.

 

 

3 modèles (4 tables) :

  • Tag
    • nom_tag: string max 30 caractères
  • Article
    • titre: string max 100 caractères
    • auteur: string max 50 caractères
    • date: datetime, par défaut rempli par la date et l’heure du serveur au moment du post de l’article. NB : les vues sont faites de telle sorte qu’un article peut être écrit à l’avance. Il ne sera visible qu’une fois que la date du serveur sera ultérieure à la date de l’article.
    • contenu: string infini
    • tag: un ou plusieurs Tag(s) (many-to-many)
    • epingler: un boolean
  • Commentaire
    • id_article: clé étrangère faisant le lien avec l’article
    • date: datetime, de la date du serveur au moment du post du commentaire.
    • contenu: string infini.

Si il y a 4 tables pour trois modèles, c’est parce que les tables Articles et Tag sont liées en many-to-many : un article peut avoir plusieurs tag et un tag peut avoir plusieurs articles. Dans ce cas, Django génère automatiquement une table intermédiaire contenant les 2 clés primaires des 2 tables concernées. Pour le développeur, cette table est totalement invisible.

b/Vue

Les vues renvoient toutes un paquet HTTP contenant la page web complète.

  • index

Il s’agit de la vue relative à l’accueil du site.

Envoie au template les 3 derniers articles en date au template index.html

 

  • actus

Prend en paramètre optionnel un nom de tag et le numéro de page.

Envoie n articles au template. Si il s’agit de la page 1, ce sont les n derniers articles. Si c’est la page 2, ce sont les n articles suivants. Il y a aussi la possibilité de choisir un tag pour ne retenir que les articles ayant ce tag.

Met également à disposition du template le nombre exact d’articles envoyés (inférieur ou égal à n) ainsi que le nombre totale de pages (nombre_articles / n).

 

  • article

Prend en paramètre “id_article”, l’identifiant de l’article à afficher, ainsi que “page_commentaire”, le numéro de page de commentaire (il y a n commentaires par page).

Les n commentaires sont fournis dans l’ordre de parution, selon la page.

La page renvoyée contient également le formulaire “CommentForm” de vitrine/forms.py. Ce formulaire contient un champ de texte infini ainsi qu’un captcha. Pour l’instant, il n’y a pas de champ utilisateur car nous n’avons pas de gestion de session.

Dans le cas de l’envoi d’un commentaire, la page la base de donnée est mise à jour, la page est raffraichie et la variable “envoi” passe à True.

 

  • festival

Prend en paramètre le numéro de la page (avec n articles par page).

Envoie au template les n derniers articles ayant les tags “festival” et/ou “concours_BD”, triées par épinglage puis par date. Un article épinglé est un article dont le champ “epingler” est égal à 1. Ceci permet de mettre en avant certains articles plus importants. Comme dans la page actus, il y a n articles ou moins par page.

 

  • Contact

Utilise le formulaire “ContactForm”, codé dans vitrine/forms.py. Ce formulaire contient 5 champs :

  • sujet: chaine de maximum 5 caractères
  • message: chaîne infinie de caractère
  • envoyeur: adresse mail
  • renvoi: boolean qui définit si l’utilisateur souhaite recevoir une copie du mail sur sa boîte mail.
  • captcha: génère un captcha en utilisant le module django-simple-captcha

L’envoi du formulaire entraine 2 choses dans la vue :

  • l’envoi du mail de contact directement à l’adresse mail du club BD/Manga
  • l’envoi optionnel d’une copie de ce mail à l’envoyeur
  • La variable booléenne “envoi” passe à True pour permettre au template d’indiquer à l’utilisateur que l’envoi s’est bien déroulé.

Fonctionnement de l’envoi de mails :

Le site dispose de sa propre adresse mail (clubbdinsa@hotmail.fr). Dans les faits, nous n’utilisons que son service d’envoie (SMTP). Les paramètres de connexions sont dans ./setup.py, ligne 142. Le mot de passe est stocké dans le fichier ./passwords.txt, fichier non partagé dans le git.

 

  • Archives

Prend en paramètre le numéro de page (avec n articles par page).

Prend également les paramètres spéciaux d’url “tag”, “year”, “month”.

Utilise le formulaire “RechercheForm” de vitrine/forms.py. Ce formulaire comporte 3 menus déroulants :

  • tag: comportant tous les tags
  • mois: comportant tous les mois où un article a été écrit
  • annee: comportant toutes les années où un article a été écrit.

La vue récupère les données fournies par l’utilisateur dans les données et se redirige vers elle-même avec les bons paramètres. Ces paramètres permettent ensuite de faire la recherche appropriée dans la base de données et de sortir les articles correspondants. Comme dans la vue actus, celle-ci envoie au template n articles par page de recherche.

 

  • Infos

Il s'agit de la vue relative à la page "infos pratiques".

Fait juste le lien avec le template “vitrine/infos.html”, qui est une page statique.

c/Templates

Afin de créer un site web structuré et cohérent visuellement, nous avons créé une page base.html qui sert de base à toutes les autres. Grâce au framework Django il est possible de faire “hériter” des caractéristiques de ce documents de référence à toutes les autres pages web du site. On utilise pour cela les balises du framework Django { % block nom_block %}. On évite ainsi de réécrire toute la page à chaque fois.

 

  • Base.html

On trouve dans ce document comme , dit ci-dessus, les éléments communs entre les différentes pages du sites. On y trouve la structure standard d’un document HTML (balises <html> <head> <body>).

  • Dans la balise <head> on retrouve des balises <link>, liens vers les fichiers css associées.
  • La balise { % load static %} sert, sur la prochaine balise { % static element %}, à charger l’élément depuis la structure du site.
  • La balise { % block styleSup %} est ici vide mais pourra être remplie par les pages intégrant base.html afin de définir d’autres fichiers CSS.
  • La balisecontient la barre de navigation du site qui est commune à toutes les pages. Les noms présents dans les balises  { % url nom_url %} font référence aux noms symboliques des urls présents dans le fichier url.py
  • La balise { % block content %} est vide et sera remplie par les pages qui “hériteront” de base.html
  • La balise <footer> contient les éléments que l’on retrouve sur toutes les pages en pied de page.

 

  • Pour le restes des pages HTML :
    • La balise { % extends "base.html" %} permet de récupérer le code html du fichier base.html
    • On définit dans {\% block styleSup %} un lien vers une feuille de style spécifique à la page.
    • On retrouve à l’intérieur de { % block  content %} le contenue de la page (variant en fonction de la section du site dans laquelle on se trouve).

 

  • L’accueil (index.html) :
    • La balise {\% for article in last_articles %} permet de récupérer les articles présents dans la variable last_articles , variable qui nous est donnée par la view de index. Pour chaque article on vient récupérer son id (pour créer le lien vers l’article en question), son titre, son contenu (que l’on tronque à 120 mots), son auteur et sa date de publication.
    • Il y a un lien “Plus d’articles”  qui redirige simplement sur la page actu.html

 

  • Actus
    • Pour trier les articles par tags, on utilise des balises { % url %} qui font appel à des urls particulières qui changent les paramètres de la vue, permettant d’afficher les bons articles.
    • Pour l’affiche des articles on utilise le même principe que index.html avec ici un affichage d’articles allant jusqu’à 10.
    • Pour naviguer entre les pages lorsqu’il y plus de 10 articles, On se sert de d’un petit code Javascript présent dans les balises <script>. Ce code crée le lien vers la page suivante et/ou précédente en fonction de la page actuel. Il calcule et affiche les numéros de page en fonction du nombre d’articles correspondant au tag.
    • En loutre, on vient récupérer les variables: page, pages_necess et nbr_articles, ce qui permet en combinaison avec la balise { % if %} d’afficher ou non “Page précédente” et “Page suivante”, si des articles existent à ces pages.

 

  • Festival
    • Cette page a la même fonction et la même construction que actus, mais affiche uniquement les articles en rapport avec le festival de BéDéologie ou le concours BD.
    • De plus elle affiche en premier les articles du festival dit “épinglés” qui sont d'actualité mais cet aspect est géré par la vue associée.

 

  • Nos ouvrages (recherche.html)
    • Cette page est à part de l’application dite “vitrine”, elle se situe dans une autre application; “bibliothèque”.
    • Elle sert de moteur de recherche, afin de trouver si un ouvrage est présent ou non dans le club.
    • Elle utilise un formulaire associé à la page (balise {{form.as_p}} qui est définit dans le fichier forms.py, qui fait le lien avec la BDD.
    • Pour le résultat de la recherche on utilise le même principe que actus.
    • Contact
    • On utilise ici un formulaire, mais contrairement à “nos ouvrages”, il est écrit directement dans le fichier HTML afin d’avoir un meilleur contrôle dessus en CSS pour le rendre plus esthétique.
    • Le bouton “Envoyer” va déclencher une requête que la view va traiter pour envoyer l’email au club BD/Mangas.
    • De plus, une fois ce traitement effectué, la variable “envoi” change, ce qui affiche à l’utilisateur que le message à bien été envoyé.

 

  • Archives
    • Cela suit exactement le même schéma que “Nos ouvrages”, mais on répertorie ici non pas les ouvrages, mais tous les articles publiés par le club.

 

  • Infos pratiques
    • Page HTML basique contenant uniquement quelques informations concernant le club.
